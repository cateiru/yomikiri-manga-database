# 実装プラン 002: DB スキーマ（packages/db）

## 概要

Drizzle ORM のスキーマ・マイグレーション・seed を `packages/db` に実装する。
web / batch の両方から共有されるデータ層の基盤となる。

## 実装タスク

### 1. パッケージセットアップ

- `drizzle-orm` / `drizzle-kit` を導入する
- `drizzle.config.ts` を作成する（接続文字列は環境変数 `DATABASE_URL`）
- ローカル開発では Docker Compose の PostgreSQL（プラン 007）を使用する
- Neon にプロジェクトを作成し、本番用の接続文字列を用意する

### 2. スキーマ定義

仕様書 5 章の通りに 3 テーブルを定義する。

- `oneshots`: 読み切り漫画
  - unique 制約 `(source_key, viewer_url)` を必ず付与する（URL 収集バッチの upsert キー）
  - `title` はビューワーページから詳細取得するまで `null` を許容する
  - `year`（`published_at` の年、年別表示用）、
    `details_fetched_at`（詳細取得バッチのキュー判定用。`null` = 未取得）を持つ
- `genres`: ジャンルマスタ（`key` に unique 制約）
- `genre_votes`: ジャンル投票
  - FK: `oneshot_id` → `oneshots.id`、`genre_id` → `genres.id`
  - unique 制約 `(oneshot_id, genre_id, anonymous_user_id)`

集計クエリ（`genre_votes` の `GROUP BY`）と一覧ソートを想定し、
`genre_votes.oneshot_id`・`oneshots.published_at`・`oneshots.details_fetched_at`
（詳細取得バッチのキュー検索用）にインデックスを検討する。

### 3. DB クライアントの使い分け

環境ごとにドライバを切り替えられるよう、entry point を分けてエクスポートする。

| 利用元 | ドライバ | export 例 |
| --- | --- | --- |
| apps/web（Workers） | `@neondatabase/serverless` | `@yomikiri/db/client-serverless` |
| apps/batch（VPS） | `postgres`（TCP） | `@yomikiri/db/client-node` |

スキーマ定義本体（`@yomikiri/db/schema`）は両者で共有する。

### 4. マイグレーション・seed

- `drizzle-kit generate` で SQL マイグレーションを生成し、リポジトリにコミットする
- `pnpm --filter @yomikiri/db migrate` で適用できるスクリプトを用意する
- seed スクリプトでジャンルマスタ 10 件を投入する（再実行しても安全な upsert 形式）

初期ジャンル（`sort_order` 順）:

1. バトル（`battle`）
2. 恋愛（`romance`）
3. コメディ・ギャグ（`comedy`）
4. ホラー（`horror`）
5. SF（`sf`）
6. ファンタジー（`fantasy`）
7. ミステリー・サスペンス（`mystery`）
8. 日常（`slice-of-life`）
9. スポーツ（`sports`）
10. ヒューマンドラマ（`drama`）

## 完了条件

- マイグレーションを新規 DB に適用すると 3 テーブルと制約がすべて作成される
- seed を実行するとジャンル 10 件が投入され、再実行してもエラー・重複が発生しない
- serverless / node 両方のクライアント経由で `select` が成功する
