## 実装タスク

### 1. Cloudflare Workers（apps/web）

- `wrangler.jsonc` を本番用に整備する（プラン 004 で作成済みの設定を拡張）
- `DATABASE_URL` を `wrangler secret put` で登録する
- `opennextjs-cloudflare build && wrangler deploy` によるデプロイスクリプトを
  `package.json` に定義する
- ISR（`revalidate: 300`）が Workers 上で機能することを確認する
- カスタムドメインの割り当て（用意できる場合）

### 2. さくらVPS（apps/batch）

- Node.js（`.node-version` と同一バージョン）と pnpm をセットアップする
- デプロイ方法: VPS 上で `git pull` + `pnpm install` + ビルドを行う
  簡易スクリプト（`deploy.sh` 等）を用意する
- `DATABASE_URL` は環境ファイル（`EnvironmentFile=`、パーミッション 600）で管理する
- systemd unit を作成する
  - `yomikiri-batch.service`: `Type=oneshot` でバッチを実行
  - `yomikiri-batch.timer`: `OnCalendar` で 6 時間ごとに起動、`Persistent=true`
- 失敗検知: 終了コード非 0 の場合に `OnFailure=` で通知 unit を起動する
  （通知先はメール等、運用に合わせて選定）
- ログは journald で確認できるようにする（`journalctl -u yomikiri-batch`）

### 3. CI（GitHub Actions）

- Pull Request / main push で以下を実行する
  - `pnpm lint`
  - `pnpm typecheck`
  - `pnpm test`（batch パーサー・読了判定などの単体テスト）
  - `pnpm build`（web の OpenNext ビルドを含む）
- main への push で Cloudflare へ自動デプロイする
  （`CLOUDFLARE_API_TOKEN` を GitHub Secrets で管理）
- マイグレーションの適用は当面手動運用とし、手順を README に記載する

### 4. 運用ドキュメント

README（またはこのディレクトリ配下）に以下をまとめる。

- ローカル開発のセットアップ手順（環境変数・DB 準備・seed）
- マイグレーションの適用手順
- VPS のバッチ配備・更新手順
- 障害時の確認ポイント（journald ログ、Neon ダッシュボード、Workers ログ）

## 完了条件

- 本番 URL でトップページが表示され、投票まで一連の機能が動作する
- VPS 上で timer が 6 時間ごとにバッチを起動し、`oneshots` が更新される
- バッチ失敗時に通知が届く
- CI が PR で lint / typecheck / test / build を検証し、main で自動デプロイされる
