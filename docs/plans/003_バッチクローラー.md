# 実装プラン 003: バッチクローラー（apps/batch）

## 概要

`sources.json` に基づき GigaViewer 系サイトの読み切りページからビューワー URL を収集し、
ビューワーページから詳細（タイトル・作者・サムネイル・掲載日）を取得して
`oneshots` テーブルへ反映するバッチを実装する。

処理は「URL 収集バッチ」「詳細取得バッチ」の 2 段に分かれる
（1 サービスだけ処理が偏り続けることを防ぐため。詳細は
[001_plan.md 8 章「バッチ仕様」](../001_plan.md) 参照）。

- 参照: [001_plan.md 4 章「対象サイト定義」](../001_plan.md)、[8 章「バッチ仕様」](../001_plan.md)

## 依存関係

- 依存する先行プラン: [002 DB スキーマ](./002_DBスキーマ.md)
- 後続プラン: 004（Web 一覧が表示するデータを供給）、006（VPS への配備）

## スコープ

| 項目 | 対象 |
| --- | --- |
| sources.json の読み込み・バリデーション | ✅ 対象 |
| gigaviewer 一覧ページパーサー（ビューワー URL 抽出）の実装 | ✅ 対象 |
| gigaviewer ビューワー詳細パーサーの実装 | ✅ 対象 |
| oneshots への upsert（URL 収集・詳細取得の 2 段） | ✅ 対象 |

| systemd timer への配備 | ❌ 対象外（プラン 006） |

## 実装タスク

### 1. sources.json ローダー

- リポジトリ直下の `sources.json` を読み込み、zod 等でバリデーションする
- `enabled: true` のソースのみを処理対象とする
- 未知の `parser` 値はエラーとして起動時に検出する

### 2. gigaviewer 一覧ページパーサー（URL 収集）

- GigaViewer で共通化されているのはビューワー・履歴ページの機構のみで、
  トップページ・作品一覧ページの HTML 構造は掲載元（出版社・レーベル）ごとに異なる。
  そのため `parser: "gigaviewer"` は「GigaViewer 採用サービス」という分類にすぎず、
  一覧ページの抽出処理は `source.key` 単位の実装が必要になる
- `parser` 種別ごとの実装を差し替えられる interface を定義する

  ```ts
  interface Parser {
    parse(html: string, source: Source): ParsedOneshotUrl[];
  }
  ```

- 一覧ページから抽出するのは **ビューワー URL のみ**とする
  （タイトル・作者・サムネイル・掲載日はビューワーページから取得するため、
  一覧ページ側では抽出しない）
- 一覧ページ内に連載・完結・読み切り等が混在するソースがある場合、該当セクションのみを抽出する
  （`listUrls` にアンカー付き URL を指定しているソースは、この混在を想定したもの。
  セクションを絞り込むセレクタは維持し、変更するのは「各アイテムから何を取り出すか」のみ）
- 1 ソースが複数の一覧ページを持つ場合は `listUrls` に複数 URL を列挙する
  （例: 通常の読み切り一覧ページと新人賞・漫画賞ページを両方対象にする場合）
- 姉妹サイト等で同一作品が重複掲載されるソースには `fallbackSourceKey` を指定できる。
  指定した source.key に既に同一パス（クエリ除く）の viewer URL が登録済みであれば、
  このソースでの登録をスキップする（重複排除ロジックは URL 収集バッチ側で実装する）
- 未対応の `source.key` はエラーとして起動時に検出する
- HTML パースには `cheerio` を使用する
- `sources.json` に登録した**ソースごと**に実ページ HTML をフィクスチャとして保存し、
  パーサーの単体テストを書く（コミックDAYS に限定しない）
  - 新規ソース追加時は、対応する抽出処理とフィクスチャの追加をセットで行う
  - 必須項目（ビューワー URL）が抽出できない要素はスキップし、警告ログを出す

### 3. gigaviewer ビューワー詳細パーサー

- GigaViewer はビューワーページのデザインが全サービス共通のため、`source.key` に依存しない
  単一の抽出関数として実装する
- 抽出項目とセレクタ

  | 項目       | セレクタ                   | 備考                                                       |
  | ---------- | --------------------------- | ------------------------------------------------------------ |
  | タイトル   | `.series-header-title`      |                                                              |
  | 作者       | `.series-header-author`     | 内部に `<a>` を含む場合はテキストのみ取得                   |
  | サムネイル | `img.series-header-image`   |                                                              |
  | 掲載日     | `.episode-header-date`      | 日付としてパースし `published_at` に格納。年を `year` にも保存 |

- タイトルが抽出できない場合は抽出失敗として扱う
- 共通フィクスチャ 1 枚（`test/fixtures/gigaviewer-viewer.html`）に対する単体テストを書く

### 4. URL 収集バッチ（クロール実行部）

- ソースを直列に処理し、リクエスト間隔を 1 秒以上空ける
- `robots.txt` を取得・解釈し、`listUrls` のいずれかが拒否されている場合は
  例外を投げてソース単位の catch まで伝播させ、当該ソース全体を今回の収集失敗として扱う
  （同じソースの他の `listUrls` が取得済みであっても upsert されない点に注意）
- User-Agent にサービス名と連絡先を設定する
  （例: `yomikiri-manga-database/1.0 (+https://github.com/cateiru/yomikiri-manga-database)`）
- HTTP エラー・タイムアウトはソース単位で catch し、他ソースの処理を継続する
- `(source_key, viewer_url)` をキーに upsert する
  - 新規: `first_seen_at` / `last_seen_at` に現在時刻を設定し、詳細情報（`title` 等）は `null` のまま
  - 既存: `last_seen_at` のみ更新する（詳細情報は上書きしない）

### 5. 詳細取得バッチ（キュー処理）

- `details_fetched_at IS NULL` の行を `source_key` ごとにグルーピングしてキューとする
- **ラウンドロビン**で各ソースのキューから 1 件ずつ処理し、1 ソースの滞留件数が多くても
  他ソースの処理が後回しにならないようにする
- レート制限（ソースごと 1 req/sec 以上の間隔）はソースごとの直近アクセス時刻で管理する
- ビューワー URL に対しても `robots.txt` の許可判定を行う
- 抽出成功: `title` / `author` / `thumbnail_url` / `published_at` / `year` /
  `details_fetched_at` を更新
- 抽出失敗（タイトル抽出不可）: `details_fetched_at` のみ更新し、無限リトライを防ぐ
- HTTP/ネットワークエラー: `details_fetched_at` を更新せず、次回バッチでの再試行対象として残す

### 6. ログ・実行結果

- 実行ごとに URL 収集・詳細取得それぞれ、ソース単位で結果を構造化ログに出力する
  - URL 収集: 取得件数・新規件数・エラー
  - 詳細取得: 試行件数・取得件数・失敗件数・エラー
- いずれかのソースが失敗した場合、プロセスの終了コードを非 0 にする
  （systemd 側で失敗を検知できるようにする）

## 完了条件

- `docker compose run --rm batch` でローカルからバッチを実行できる
  （内部で URL 収集 → 詳細取得の順に実行される）
- 実行後、ローカル DB（Docker Compose の PostgreSQL）の `oneshots` に
  `sources.json` の `enabled: true` な全ソースのビューワー URL が登録され、
  詳細（タイトル等）が取得されている
- 2 回連続で実行しても重複行が発生せず、既存行は `last_seen_at` のみ更新される
- 一度詳細を取得した行は再取得されない（`details_fetched_at` が設定され続ける限りキュー対象外）
- ソースごとのフィクスチャに対する一覧ページパーサーの単体テスト、および
  ビューワー詳細パーサーの単体テストが全て通る
