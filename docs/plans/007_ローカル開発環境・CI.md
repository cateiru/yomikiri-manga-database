# 実装プラン 007: ローカル開発環境・CI

## 概要

Docker Compose によるローカル開発環境と、GitHub Actions による CI を整備する。
デプロイワークフロー（CD）は本プランでは扱わず、[006 デプロイ・運用](./006_デプロイ・運用.md) で追加する。

- 参照: [001_plan.md 12 章「ローカル開発環境」](../001_plan.md)、[13 章「CI」](../001_plan.md)

## 依存関係

- 後続プラン: 002 以降のローカル開発・動作確認はこの環境の上で行う

## スコープ

| 項目 | 対象 |
| --- | --- |
| compose.yaml（db / web / wsproxy / batch） | ✅ 対象 |
| CI ワークフロー（lint / typecheck / test / build） | ✅ 対象 |

| 本番インフラ（Workers / VPS） | ❌ 対象外（プラン 006） |

## 実装タスク

### 1. compose.yaml

リポジトリ直下に `compose.yaml` を作成し、4 サービスを定義する。

| サービス | ベースイメージ | 内容 |
| --- | --- | --- |
| `db` | `postgres` 公式イメージ | ポート 5432 を公開。named volume でデータを永続化 |
| `web` | `node`（`.node-version` と同一バージョン） | リポジトリを bind mount し、`pnpm install` 後に `pnpm --filter @yomikiri/web dev` を実行。ポート 3000 を公開 |
| `wsproxy` | WebSocket ⇔ TCP 変換用イメージ | `web` が本番と同じ `@neondatabase/serverless`（WebSocket 経由）ドライバをそのまま使えるよう、`db` への通常 TCP 接続へ中継する |
| `batch` | `node`（`.node-version` と同一バージョン） | `pnpm --filter @yomikiri/batch start` を実行する one-shot。`profiles: ["batch"]` で `docker compose up` の対象から除外 |

- `web` / `batch` の `DATABASE_URL` は `db` サービス（`web` からは `wsproxy` 経由）を指す値を compose 内で設定する
- `node_modules` は named volume でコンテナ内に分離し、ホスト側と混在させない
- `db` に healthcheck を設定し、`web` / `batch` は
  `depends_on: condition: service_healthy` で DB 起動後に開始する
- `web` コンテナは起動のたびに `apps/web/.dev.vars` を `wsproxy` 経由の接続情報で上書きする。
  Docker を使わずホストで直接 `next dev` する場合は、この上書きが行われない点に注意する

### 2. 起動手順の整備

- `docker compose up` で db + web が起動し、`http://localhost:3000` を確認できるようにする
- バッチは `docker compose run --rm batch` で手動実行する
- 初回セットアップ手順（migrate / seed をコンテナ経由で実行するコマンド）を README に記載する

### 3. CI（.github/workflows/ci.yml）

- トリガー: Pull Request と main への push
- 実行内容: `pnpm lint` / `pnpm typecheck` / `pnpm test` / `pnpm build`
  （`test` はプラン 003 / 005 で単体テストが追加されるまでは空実行を許容する）
- セットアップ: `actions/setup-node`（`.node-version` を参照）+ pnpm + store キャッシュ
- デプロイは行わない（プラン 006 で別ワークフローとして追加する）

## 完了条件

- `docker compose up` で db / web が起動し、トップページにアクセスできる
- `docker compose run --rm batch` でバッチを実行できる
- コンテナを破棄・再作成しても volume により DB データが保持される
- PR 作成時に CI が lint / typecheck / test / build を実行し、成功する
