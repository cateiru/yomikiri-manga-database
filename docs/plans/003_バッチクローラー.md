# 実装プラン 003: バッチクローラー（apps/batch）

## 概要

`sources.json` に基づき GigaViewer 系サイトの読み切り一覧を取得し、
`oneshots` テーブルへ upsert するバッチを実装する。

- 参照: [001_plan.md 4 章「対象サイト定義」](../001_plan.md)、[8 章「バッチ仕様」](../001_plan.md)

## 依存関係

- 依存する先行プラン: [002 DB スキーマ](./002_DBスキーマ.md)
- 後続プラン: 004（Web 一覧が表示するデータを供給）、006（VPS への配備）

## スコープ

| 項目 | 対象 |
| --- | --- |
| sources.json の読み込み・バリデーション | ✅ 対象 |
| gigaviewer パーサーの実装 | ✅ 対象 |
| oneshots への upsert | ✅ 対象 |
| クロールマナー（robots.txt / UA / 間隔） | ✅ 対象 |
| systemd timer への配備 | ❌ 対象外（プラン 006） |

## 実装タスク

### 1. sources.json ローダー

- リポジトリ直下の `sources.json` を読み込み、zod 等でバリデーションする
- `enabled: true` のソースのみを処理対象とする
- 未知の `parser` 値はエラーとして起動時に検出する

### 2. gigaviewer パーサー

- `parser` 種別ごとの実装を差し替えられる interface を定義する

  ```ts
  interface Parser {
    parse(html: string, source: Source): ParsedOneshot[];
  }
  ```

- `gigaviewer` パーサーを実装し、一覧ページの HTML から次の項目を抽出する
  - タイトル（必須） / 作者 / サムネイル URL / ビューワー URL（必須） / 掲載日時
- HTML パースには `cheerio` 等の軽量ライブラリを使用する
- コミックDAYS の実ページ HTML をフィクスチャとして保存し、パーサーの単体テストを書く
  - 必須項目が抽出できない要素はスキップし、警告ログを出す

### 3. クロール実行部

- ソースを直列に処理し、リクエスト間隔を 1 秒以上空ける
- `robots.txt` を取得・解釈し、`listUrl` が拒否されている場合はスキップする
- User-Agent にサービス名と連絡先を設定する
  （例: `yomikiri-manga-database/1.0 (+https://github.com/cateiru/yomikiri-manga-database)`）
- HTTP エラー・タイムアウトはソース単位で catch し、他ソースの処理を継続する

### 4. upsert 処理

- `(source_key, viewer_url)` をキーに `ON CONFLICT` で upsert する
  - 新規: `first_seen_at` / `last_seen_at` に現在時刻を設定
  - 既存: タイトル等のメタデータと `last_seen_at` のみ更新（`first_seen_at` は保持）
- 一覧から消えた作品は削除しない

### 5. ログ・実行結果

- 実行ごとにソース単位で「取得件数 / 新規件数 / 更新件数 / エラー」を構造化ログで出力する
- いずれかのソースが失敗した場合、プロセスの終了コードを非 0 にする
  （systemd 側で失敗を検知できるようにする）

## 完了条件

- `pnpm --filter @yomikiri/batch start` でローカルからバッチを実行できる
- 実行後、Neon の `oneshots` にコミックDAYS の読み切りが登録されている
- 2 回連続で実行しても重複行が発生せず、`last_seen_at` のみ更新される
- パーサーの単体テスト（フィクスチャベース）が通る
